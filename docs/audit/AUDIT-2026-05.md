### Findings at a glance

**FAIL (must address before submission):**
1. **F-1.** `torch.use_deterministic_algorithms(True)` is **never called**, despite explicit claims in `PLAN.md § 11.1` and `PRD_training_evaluation.md § 5.3` that `shared/seeding.py` calls it. NFR-2 (device-agnostic determinism) is therefore weaker than documented.
2. **F-2.** `tests/integration/test_reproducibility.py` is the only integration test claimed by the PRDs to enforce determinism (T-IT-02 / NFR-2), and it does NOT call `use_deterministic_algorithms`. The reproducibility net is single-platform CPU only — the documented "device-agnostic determinism" claim is unverified by tests.
3. **F-3.** `docs/TODO.md` M2a claims signal_gen.py is "103 code-LOC (excl. docstrings), 145 incl." — actual measurement is **120 code-LOC excl. docstrings** (still ≤ 150, so no LOC violation, but the in-repo claim is wrong by 17 lines and needs reconciliation before someone audits the docs against grep).

**SUSPICIOUS (warrant explicit acknowledgement before README):**
- **S-1.** Train/test contamination is large in absolute terms (1,885 of 3,549 unique test (t₀,k) pairs — **53 %** — appear identically in train; 95 % of unique test t₀ start indices appear in train). ADR-016 explicitly accepts this on stationarity grounds, but the README narrative must reproduce ADR-016's framing sentence (AC-DS-9) and the per-frequency MSE caveat in PRD § 7.3 — otherwise the lecturer will read the test split as a clean held-out and find the overlap independently.
- **S-2.** The audit prompt's hypothesis ("2 Hz target is near-constant within a window → model needs hidden-state info") is **not what the data shows**. Per-k target variance at β=π/4 is uniformly ≈ 0.5 across all four frequencies (target sweeps the full sinusoid range because t₀ ranges over ~20 full periods at 2 Hz). The 2 Hz difficulty needs a different mechanistic story before it goes into the README.
- **S-3.** Pearson r between input-window and target-window is highest at 10 Hz (+0.40) but 10 Hz has the **highest** per-frequency MSE — the simple "more correlation → easier" story does not explain the inverted profile and must not be the README's mechanistic claim.
- **S-4.** `_EVAL_BATCH_SIZE = 256` is a literal in `services/evaluation.py:28`. Doesn't change what's computed (loss is averaged), but technically violates guidelines § 6.2 ("All configurable values MUST come from configuration files").
- **S-5.** `tests/unit/test_training.py:137-140` uses a clever-but-confusing `(tmp_path / "a").mkdir() or True` side-effect-in-conditional pattern. Functionally correct, but a future reader will be confused.
- **S-6.** EXP-003 / EXP-008 LSTM-vs-RNN improvement is **+0.81 % to +2.48 %** — far below the PRD § 8.3 "10 % practical significance threshold". Sign-preservation across all 4 cells is real but the README must NOT overclaim a "thesis confirmation" — Outcome C (refutation by PRD criteria) is the correct verdict on EXP-003 alone.
- **S-7.** Spearman ρ at β=π/4 is **+0.80 (p=0.958)** — *opposite of thesis direction*. EXP-003 doc records this honestly; the README must echo the same statement and not paper over it.
- **S-8.** `shared/config.py` uses `eval()` (line 59) on angle expressions. The whitelist regex `^[\d.\*\/\+\-\(\)\s]+$` does mitigate, and the docstring justifies the boundary; still worth a note since it's the single `eval` call in the project.

**SKIPPED:**
- **K-1.** Coverage type breakdown (statement vs branch vs path). `pyproject.toml` has `fail_under = 85` and the run reports "Cover" only; branch coverage is not separately reported. Statement coverage is 100 %. Branch coverage was not measured.

### Severity triage

- **BLOCKERS:** F-1, F-2, S-6, S-7. F-1/F-2 are doc-vs-code lies that the lecturer will catch. S-6/S-7 are correct in `docs/experiments/EXP-003-*.md` but if the README reads "thesis confirmed" they become misleading by composition.
- **MEDIUM:** F-3, S-1, S-2, S-3 — narrative-/doc-correctness; will degrade the analysis chapter if not fixed.
- **COSMETIC:** S-4, S-5, S-8, K-1.

### Recommendation

**Project is in submittable shape on the engineering axes** (143 tests pass; 100 % statement coverage; ruff clean; no file > 150 code-LOC; no external API; uv-only; correct architecture; correct param counts; correct MSE formula; checkpoints saved/restored in the right order). **The submission risk is in the narrative**: PLAN/PRD claim a determinism call that the code does not make (F-1/F-2), and the lecturer's thesis is *refuted at the 10 % threshold by the project's own pre-registered criteria* (S-6/S-7). The README must report Outcome C honestly, with the +0.8 to +2.5 % LSTM advantage and the inverted Spearman ρ shown plainly. **No experiments need to be re-run.** The fixes are: (a) add the `torch.use_deterministic_algorithms` call (or amend PLAN/PRD if the omission is intentional, with a new ADR justifying the omission); (b) write Outcome-C-honest README sections; (c) reconcile the TODO LOC claim. Estimated time to address: ~2 hours of writing + a tiny code/test patch for F-1.
